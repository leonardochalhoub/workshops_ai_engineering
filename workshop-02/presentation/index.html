Módulo 00

Spec vence vibe

Semana passada provamos que a IA digita qualquer coisa. Esta semana a lição vira: a qualidade do que você lança é a qualidade do que você pede. O spec É o prompt.

Vibe coding

“Só me faz um app”

  • A IA adivinha o que você quis dizer
  • Re-prompt, refaz, desvia, incha

Spec-Driven Development

Decidir primeiro, depois gerar

  • Um spec escrito que a IA não interpreta errado
  • Rastreável: ideia → requisito → código
  • Uma fonte da verdade, cinco construtores alinhados

O método

Cinco fases, da ideia ao lançamento

O AgentSpec roda um fluxo SDD de 5 fases. Cada fase é um comando e gera um documento que a fase seguinte usa. Hoje vivemos nas fases 1 e 2.

Uma regra para avançar
O DEFINE precisa atingir clareza ≥ 12/15 antes do DESIGN. Se o spec está vago, você não constrói — esse é o objetivo.

Módulo 02 · decisão do dia

Como os cinco vão construir?

O mesmo kit AgentSpec, três formas de apontá-lo. Escolhemos juntos, hoje — isso define o formato das Semanas 3 e 4.

Três caminhos

Solo · Equipe · Ambos

🧑‍💻
A · Solo
Cada um dos 5 constrói o SEU protótipo. Cinco ideias, cinco repos, cinco URLs no ar.
máxima autoria
👥
B · Equipe
Os 5 constroem UM app juntos, com papéis. Um repo, produto mais profundo, trabalho em equipe de verdade.
máxima profundidade
🔀
C · Ambos (híbrido)
Cada um lança um protótipo solo pequeno hoje → o grupo escolhe UM para levar até o fim.
recomendado

Trade-offs honestos

O que cada caminho custa e entrega

Melhor quando…as ideias são bem diferentesuma ideia anima todosvocê quer os dois, em ordem

Minha recomendação

Começar solo, depois convergir

1
Hoje — cinco protótipos solo minúsculos
Cada um roda /define + /design na sua ideia e sobe um esqueleto. Máxima prática, máximo aprendizado, ninguém é passageiro.
2
Fim de hoje — o grupo vota
Cinco pitches de 60 segundos. Escolham o ÚNICO app com o spec mais forte para levar às Semanas 3–4 em equipe.
3
Semanas 3–4 — equipe no vencedor
Papéis atribuídos (PM · dados · frontend · QA · design). Os outros quatro protótipos ficam no portfólio de cada um.
💡
Por que o híbrido vence aqui
Largura primeiro (todos constroem), profundidade depois (um vai longe). Cinco portfólios E um produto de equipe forte — e a escolha é conquistada por um spec real, não por levantar a mão.
03

Módulo 03 · Fase 1

DEFINE — fixar o quê

Antes de qualquer tela ou tabela: quem sente a dor, como é o sucesso e como saberemos que funciona. Um comando escreve o doc com você.

Anatomia · DEFINE_{app}.md

Seis caixas para preencher

🎯
Problema
Quem sente a dor e o custo dela — em 1–2 frases.
👤
Usuários-alvo
2 personas, cada uma com papel e frustração.
📊
Metas · MoSCoW
MUST / SHOULD / COULD. MUST = o MVP morre sem isso.
📐
Critérios de sucesso
Mensuráveis, com números. “Lançar uma transação em < 10s.”
✅
Testes de aceitação
Dado / Quando / Então — caminho feliz, erro, borda.
🚫
Fora do escopo
O que deliberadamente NÃO vamos construir agora. Protege o MVP.

A disciplina

Ache o ÚNICO fluxo central

Um MVP é um fluxo bem feito, não dez pela metade. Escreva como uma única user story — essa frase vira a espinha de todo o build.

user-story.md
# O formato de user story Como <tipo de usuário> quero <fazer uma coisa> para que <eu obtenha este valor>.
